# smart-esb

연계(ESB) 트랜잭션을 모니터링하고, 챗봇에게 물어보면 현황을 요약해주는 대시보드입니다.
지금은 모니터링 + 챗봇만 있고, 연계 실행 기능(Flow/템플릿)은 잠시 빼두고 하나씩 추가할 예정입니다.
**AI는 전혀 쓰지 않습니다** — 챗봇도 실행 로그를 SQL로 집계한 평문 요약으로만 답합니다.

## 빠른 시작

```bash
npm install
cp .env.example .env
npm run seed:demo   # 더미 트랜잭션을 채워서 화면을 바로 확인
npm run dev         # http://localhost:4000
```

브라우저에서 `http://localhost:4000` 접속 → **모니터링** 탭에서 트랜잭션 목록, **커넥터
모니터링** 탭에서 큐/HTTP/DB/FILE별 처리 현황, **통계** 탭에서 일자별/타입별/인터페이스별
그래프, **챗봇** 탭에서 "금일 연계 에러 현황 정리해줘" 같은 질문을 해보세요.

## 모니터링 화면

트랜잭션 목록은 다음 컬럼으로 표시됩니다(모두 우측 정렬): **트랜잭션ID, 인터페이스명(인터페이스ID),
연계 시작시간, 연계 종료시간, 연계 건수(실패/전체, 예: `0/4`, `3/4`), 결과(성공/부분실패/실패),
에러내용, 소요시간**. 목록은 페이지네이션되어 있고(기본 20건/페이지), 행을 클릭하면 해당
트랜잭션에 포함된 **레코드별 상세 데이터**(실제 payload, 개별 성공/실패 여부, 에러 메시지)를
볼 수 있습니다. 실패한 레코드에는 **재전송** 버튼이 있어 클릭하면 해당 레코드만 재처리되고
(이 데모 환경은 실제 목적지가 없어 재전송은 항상 성공으로 처리됩니다), 트랜잭션의 결과·건수·
에러내용이 다시 계산됩니다.

## 커넥터 모니터링 화면

인터페이스가 커넥터 타입별로 수천~수만 개로 늘어날 수 있다는 전제로, 통계 화면과 동일하게
**서브메뉴 4개(HTTP / QUEUE / DB / FILE)**로 나누어져 있고, 각 서브메뉴는 페이지네이션
(`LIMIT`/`OFFSET`)으로 필요한 만큼만 불러옵니다. **검색창은 소메뉴마다 따로 있지 않고 공통**이라서,
검색어를 입력하면 현재 선택된 소메뉴(타입)와 상관없이 모든 커넥터 타입에서 이름을 찾습니다(카드마다
실제 타입에 맞는 필드가 그대로 표시됨). 검색어를 지우면 다시 선택된 소메뉴의 타입으로만 좁혀서
보여줍니다. 타입별로 다른 정보를 보여줍니다:

- **QUEUE**: 출발지 → 도착지, 큐 이름, 상태(정상/지연), 적체 건수, 최고 적체 시간(가장 오래
  쌓여있는 실패 레코드의 경과 시간). 적체 건수는 "오늘 실패해서 아직 재전송되지 않은 레코드 수"로
  계산되므로, 모니터링 탭에서 레코드를 재전송하면 여기서도 즉시 줄어듭니다.
- **HTTP**: 호출 주소(메서드+URL), 서비스 IP, 타임아웃(ms), 느린 호출 비율(타임아웃을 넘긴
  트랜잭션의 비율)
- **DB** / **FILE**: 테이블명/워터마크 컬럼(DB) 또는 파일 경로(FILE), 폴링 주기, 스케줄
  상태(정상/지연) — 폴링 주기 대비 마지막 실행 시각이 너무 오래됐으면(폴링 주기의 1.5배 초과)
  "지연"으로 표시합니다.

공통으로 오늘 처리 건수(성공/실패/실패율), 소요시간(평균/최소/최대), 마지막 처리 시각, 최근
에러 메시지(최대 3건, 중복 제거)도 함께 보여줍니다. 인터페이스 목록/타입/설정은 `interfaces`
테이블에 등록되어 있고(현재는 `npm run seed:demo`/`npm run seed:scale`로만 채워짐, 등록/수정
화면은 다음 단계 예정), 실제 수치는 `interface_runs`의 오늘자 데이터를 집계해서 만듭니다.

## 통계 화면

일자/커넥터 타입/인터페이스가 각각 다르게 늘어날 수 있다는 전제로, **서브메뉴 3개**로
나누어져 있고 각각 규모에 맞게 설계했습니다:

- **일자별**: 7/30/90/180일 프리셋으로 기간을 고를 수 있습니다. 31일을 넘으면 자동으로
  7일 단위 주간 버킷으로 묶어서, 기간이 아무리 길어져도(예: 1년) 막대가 항상 30개 안쪽으로
  유지됩니다. 버킷이 많을 때는 x축 라벨도 8개 정도로만 성기게 표시해서 겹치지 않게 합니다.
- **연계 타입별**: `QUEUE`/`HTTP`/`DB`/`FILE`로 종류가 고정돼 있어 인터페이스가 아무리
  늘어나도 항목 수는 늘지 않습니다.
- **인터페이스별**: 인터페이스가 수천~수만 개로 늘어날 수 있으므로, 전체를 한 번에 내려주지
  않고 이름 검색 + 페이지네이션(SQL의 `LIMIT/OFFSET`과 `LIKE`)으로 필요한 만큼만 불러옵니다.
  `npm run seed:scale`로 인터페이스 3,000개를 만들어서 실제로 검색·페이지 이동이 빠르게
  동작하는지 확인해볼 수 있습니다.

그래프는 외부 차트 라이브러리 없이 순수 HTML/CSS로 그리며, 각 그래프 아래에 동일한 수치를
표(테이블)로도 제공합니다. 집계는 모두 SQL이 직접 하므로 여기에도 AI는 쓰이지 않습니다.

## 챗봇 동작 방식

질문에서 "오늘/금일", "어제", "이번주", "최근 N시간", "최근 N일" 같은 표현을 정규식으로 감지해
조회 기간을 정하고, 그 기간의 트랜잭션을 SQL로 집계해서 고정된 형식의 문장으로 답합니다.
AI를 전혀 호출하지 않으므로 항상 같은 입력에는 같은 답이 나옵니다.

## API

| Method | Path | 설명 |
|--------|------|------|
| GET | `/api/monitoring/runs?page=&pageSize=&interfaceId=` | 트랜잭션 목록 (페이지네이션) |
| GET | `/api/monitoring/runs/:transactionId` | 트랜잭션 상세 (레코드별 payload/상태/에러 포함) |
| POST | `/api/monitoring/runs/:transactionId/records/:recordId/resend` | 실패한 레코드 재전송 (결과/건수/에러내용 재계산) |
| GET | `/api/monitoring/summary?from=&to=` | 기간별 인터페이스 성공/부분실패/실패 집계 |
| GET | `/api/connectors?type=&page=&pageSize=&search=` | 인터페이스 목록 + 오늘 처리 현황 (페이지네이션 + 검색). `type` 생략 시 QUEUE/HTTP/DB/FILE 전체 대상 |
| GET | `/api/stats/daily?days=` | 일자별 처리 건수 (31일 초과 시 자동 주간 집계) |
| GET | `/api/stats/by-type?days=` | 커넥터 타입별 처리 건수 |
| GET | `/api/stats/by-interface?days=&page=&pageSize=&search=` | 인터페이스별 처리 건수 (페이지네이션 + 검색) |
| POST | `/api/chat` | `{ "question": "..." }` → 실행 로그 기반 평문 요약 답변 |
| GET | `/health` | 헬스체크 |

## 데이터 저장

트랜잭션은 `data/smart-esb.sqlite`(SQLite, WAL 모드) `interface_runs` 테이블에 저장됩니다.
레코드별 상세(payload, 개별 성공/실패, 에러)는 같은 행의 `records_json` 컬럼에 JSON으로
저장되어 상세 조회 시에만 함께 내려줍니다. `.env`의 `DATA_DIR`로 위치를 바꿀 수 있고, 이
파일은 `.gitignore`에 포함되어 커밋되지 않습니다.

`npm run seed:demo`는 `scripts/seed-monitoring-demo.ts`를 실행해 가상의 인터페이스 4개에 대해
최근 7일치 더미 트랜잭션(성공/부분실패/실패 섞임, 레코드별 상세 데이터 포함)을 채워 넣습니다.
오늘(`daysAgo=0`) 데이터는 항상 "오늘 자정~현재 시각" 사이로만 생성되므로, 커넥터 모니터링의
"오늘 처리" 수치가 모든 인터페이스에서 채워집니다.

`npm run seed:scale`(`scripts/seed-scale-interfaces.ts`)은 통계 > 인터페이스별 화면과 커넥터
모니터링 화면의 검색/페이지네이션이 실제 대규모 환경에서도 버티는지 확인하기 위한 스트레스
테스트용 시드입니다. 기본 3,000개(`npm run seed:scale -- 10000`처럼 인자로 개수 조절 가능)의
인터페이스(QUEUE/HTTP/DB/FILE에 고르게 분배, 타입별 설정값도 함께 생성)와 트랜잭션을 한
트랜잭션(batch)으로 빠르게 삽입합니다.
